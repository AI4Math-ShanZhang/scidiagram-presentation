Answering Q1

A1: Current Models Lack Genuine Diagram Understanding

We establish this through a 4-step argument

The Core Problem

Models fail to recognize basic elements in diagrams

Models fail to parse diagrams into correct structural topology

Lack of symbolic understanding of primitives

Why Strong on Natural Images but Fail on Diagrams?

1

Diagrams are structured interfaces, not natural images for visual aesthetics

2

Code-based representations expose structural understanding

3

Perception & symbolic understanding are prerequisites for reasoning

4

Even with strong textual reasoning, models fail at perception & symbolic understanding

Let's walk through each step...

A1 · Step 1 of 4

Diagrams Are Structured Interfaces, Not Natural Images

Why diagrams require fundamentally different understanding

Designed Artifacts

Unlike photos, diagrams encode precise structural & logical relationships

Correctness is discrete and verifiable, not continuous/subjective

Small visual changes can completely alter meaning

Appearance similarity ≠ semantic equivalence

Domain Examples

Circuit Diagram

Correctness = topological connectivity, not visual aesthetics

Geometry Figure

Meaning = angle markers & segment labels, not pixel colors

Chemical Structure

Meaning = bond types & atom positions, not visual style

Implication: We need evaluation methods that test structural comprehension, not just visual similarity

A1 · Step 2 of 4

Code-Based Representations Expose Structural Understanding

Why code is better than natural language for evaluation

Why Not Natural Language?

Inherently ambiguous - same structure, many descriptions

Hard to describe spatial arrangements precisely

Difficult to evaluate - no ground truth comparison

Domain-Specific Packages

ChemFig
CircuiTikZ
TKZ-Euclide

Verifiability

Code can be executed and rendered for comparison

Precision

Structural relationships are explicitly encoded in syntax

Conciseness

Domain packages compress disciplinary knowledge into compact syntax

Key Insight: The ability to generate correct code serves as evidence of structural and topology understanding

A1 · Step 3 of 4

Perception & Symbolic Understanding Are Prerequisites for Reasoning

Scientific reasoning requires grounding on accurate perception

The Reasoning Chain

1

PARSE

Identify diagram elements & structure

2

INTERPRET

Understand domain-specific symbolic meanings

3

REASON

Derive valid scientific conclusions

Symbolic Understanding Examples

ChemFig: Hexagon → Benzene ring

Model must know organic chemistry conventions

CircuiTikZ: Arrow → Current flow direction

Model must know electrical engineering conventions

TKZ-Euclide: Rectangle marker → Right angle

Model must know geometry notation

Without semantic grounding, models cannot transfer diagram understanding to solve scientific questions

A1 · Step 4 of 4

Current Models Fail at Foundational Steps

Evidence of shallow perception, not genuine understanding

Failure 1: Misleading Metrics

High scores on FID, CLIP while failing on structural correctness. These metrics measure visual similarity, not structural precision.

Failure 2: Missing Semantic Changes

Small but significant changes (e.g., resistor → capacitor, layout/color changes) are often missed.

Failure 3: Inconsistent Interpretation

Models may generate code but fail to understand symbolic and spatial relationships among primitives.

The Perception Gap

Global Metrics (FID, CLIP)
measure visual similarity

Structural Correctness
measures logical & topological accuracy

This motivates our benchmark: How to evaluate these failures?

Answering Q2

A2: Benchmark Design - Three Solutions

How we design a benchmark to effectively evaluate these limitations

L1

Data Limitations

Limited diagram types

Synthetic images only

Narrow subject coverage

S1: Diverse Real-World Data

8 categories, 7,424 diagrams from papers & docs

L2

Metric Limitations

Global metrics miss structure

Token metrics are order-sensitive

Focus on syntax, not perception

S2: Primitive-Level Metrics

Type, Text, Color, Layout - order-invariant

L3

Evaluation Limitations

One-step evaluation only

Can't diagnose what fails

No ability decomposition

S3: Progressive Tasks

D2C → DE → DU, isolate weak abilities

Core Design: Executable code + multi-granularity evaluation (token/primitive/global) + progressive task design to isolate what ability the model lacks

A2 · Solution S2 Details

Primitive-Level Evaluation Metrics

Four semantic dimensions extracted directly from code syntax

Type

Element types with domain meaning

"resistor", "benzene", "scatter"

Text

Labels, titles, annotations

with semantic positions

Color

Draw/fill colors

component-specific

Layout

Spatial arrangement

topological connectivity

Why Primitive-Level Metrics?

Order-invariant: Equivalent code with different ordering is treated equally

Semantically meaningful: Measures actual structural understanding

Edit-sensitive: Detects instruction-conditioned modifications

Extraction Method

Primitives extracted from TikZ/LaTeX syntax

Domain-specific parsers for each package

Direct comparison without rendering

A2 · Solution S3 Details

Progressive Task Framework: D2C → DE → DU

Each task builds upon capabilities required by the previous one

D2C

Diagram-to-Code

  • Syntax Knowledge
  • Primitive Recognition
  • Primitive-to-Syntax Mapping

Output: Code

→
DE

Diagram Editing

  • All D2C abilities
  • Instruction Grounding
  • NL → code locations

Output: Edited Code

→
DU

Diagram Understanding

  • All D2C + DE abilities
  • Domain Semantics
  • Symbolic Reasoning

Output: Answer

D2C fails + valid syntax
→ Primitive recognition/grounding issue

D2C passes, DE fails
→ Instruction grounding issue

DE passes, DU fails
→ Symbolic understanding issue

Diagnostic Power: Pinpoint exactly where and what ability a model lacks
